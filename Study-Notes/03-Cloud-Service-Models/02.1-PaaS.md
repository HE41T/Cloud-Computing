# Platform as a Service (PaaS)

## PaaS คืออะไร
Platform-as-a-Service (PaaS) คือรูปแบบบริการคลาวด์ที่ให้ **สภาพแวดล้อมสำหรับรันแอปพลิเคชัน (Runtime Environment)**  
รวมถึงเครื่องมือที่จำเป็นสำหรับการ **พัฒนา (Development)** และ **ปรับใช้งาน (Deployment)** แอปพลิเคชัน

PaaS มีจุดเด่นคือเครื่องมือแบบ **Point-and-Click**  
ซึ่งช่วยให้ผู้ที่ไม่ใช่นักพัฒนา (Non-developers) สามารถสร้างเว็บแอปพลิเคชันได้

ตัวอย่างผู้ให้บริการ PaaS:
- Google App Engine
- Force.com

นักพัฒนาสามารถเข้าใช้งานแพลตฟอร์มเหล่านี้ผ่านเว็บไซต์  
และใช้ **Built-in API** ที่มีให้ เพื่อพัฒนาเว็บแอปพลิเคชันได้ทันที

---

## ข้อจำกัดของ PaaS (Vendor Lock-in)
ข้อเสียที่สำคัญของ PaaS คือ **Vendor Lock-in**

ตัวอย่าง:
- แอปพลิเคชันที่เขียนด้วยภาษา Python
- ใช้ API ของ Google
- และรันบน Google App Engine

แอปพลิเคชันลักษณะนี้มักจะทำงานได้ดีเฉพาะในสภาพแวดล้อมของผู้ให้บริการรายนั้น  
ทำให้ย้ายไปแพลตฟอร์มอื่นได้ยาก

---

## หลักการทำงานของ PaaS
PaaS จะให้:
- API
- เครื่องมือพัฒนา
- เครื่องมือ Deployment

แก่นักพัฒนา  
และช่วยให้ผู้ใช้งานปลายทาง (End User) สามารถเข้าถึงแอปพลิเคชันทางธุรกิจได้ผ่านอินเทอร์เน็ต

---

## ประโยชน์ของ PaaS (Benefits)

### 1. ภาระงานด้านการดูแลระบบต่ำ
ผู้ใช้งานไม่จำเป็นต้องดูแลระบบเอง  
เนื่องจากเป็นหน้าที่ของผู้ให้บริการคลาวด์

### 2. ลดต้นทุนรวมในการเป็นเจ้าของระบบ (TCO)
ผู้ใช้งานไม่ต้องลงทุนซื้อ:
- ฮาร์ดแวร์ราคาแพง
- เซิร์ฟเวอร์
- ระบบไฟฟ้า
- ระบบจัดเก็บข้อมูล

### 3. รองรับการขยายระบบ (Scalability)
สามารถปรับเพิ่มหรือลดทรัพยากรได้โดยอัตโนมัติ  
ตามความต้องการใช้งานจริง

### 4. ซอฟต์แวร์มีความทันสมัยอยู่เสมอ
ผู้ให้บริการคลาวด์รับผิดชอบ:
- การอัปเดตเวอร์ชันซอฟต์แวร์
- การติดตั้งแพตช์ด้านความปลอดภัย

---

## ปัญหาและข้อจำกัดของ PaaS (Issues)

เช่นเดียวกับ SaaS, PaaS ต้องพึ่งพา:
- เว็บเบราว์เซอร์ของผู้ใช้งาน
- การเชื่อมต่อเครือข่ายที่ปลอดภัยและเสถียร

ดังนั้น PaaS จึงมีปัญหาร่วมกับ SaaS  
และยังมีปัญหาเฉพาะเพิ่มเติมดังนี้

### 1. การย้ายระบบระหว่าง PaaS ทำได้ยาก
แม้จะใช้ภาษาโปรแกรมมาตรฐานเหมือนกัน  
แต่การออกแบบบริการแพลตฟอร์มอาจแตกต่างกัน

ตัวอย่าง:
- Interface ของไฟล์
- Queue
- Hash Table

ความแตกต่างเหล่านี้ทำให้การย้ายแอปพลิเคชันระหว่าง PaaS แต่ละเจ้าเป็นเรื่องยาก

---

### 2. การจัดสรรทรัพยากรแบบ Event-based
แอปพลิเคชันบน PaaS มักเป็นแบบ **Event-oriented**

ข้อจำกัด:
- แอปต้องตอบสนองคำขอภายในเวลาที่กำหนด
- มีข้อจำกัดด้านทรัพยากรและเวลาในการประมวลผล

---

### 3. วิศวกรรมความปลอดภัยของแอปพลิเคชัน
เนื่องจากแอปพลิเคชันบน PaaS พึ่งพาเครือข่ายเป็นหลัก  
จึงจำเป็นต้อง:
- ใช้เทคนิคการเข้ารหัสข้อมูล (Cryptography)
- จัดการความเสี่ยงด้านความปลอดภัยอย่างรัดกุม

---

## คุณลักษณะของ PaaS (Characteristics)

- มีสภาพแวดล้อมการพัฒนาแบบใช้งานผ่านเว็บเบราว์เซอร์
- นักพัฒนาสามารถ:
  - สร้างฐานข้อมูล
  - แก้ไขโค้ดแอปพลิเคชัน
  ผ่าน API หรือเครื่องมือแบบ Point-and-Click

- มีระบบความปลอดภัยและการปรับขยายระบบในตัว
- รองรับ Web Service Interface
- มีเครื่องมือสำหรับ:
  - Workflow
  - Approval Process
  - Business Rules

- สามารถเชื่อมต่อและทำงานร่วมกับแอปพลิเคชันอื่นบนแพลตฟอร์มเดียวกันได้ง่าย
- รองรับการเชื่อมต่อกับแอปพลิเคชันภายนอกแพลตฟอร์มผ่าน Web Services

---

## ประเภทของ PaaS (PaaS Types)

PaaS สามารถแบ่งออกเป็น 4 ประเภทหลักตามหน้าที่การทำงาน

### 1. Stand-alone Development Environments
- เป็น PaaS ที่ทำงานแบบอิสระ
- ใช้สำหรับงานเฉพาะด้าน
- ไม่มีการผูกกับ SaaS หรือไลเซนส์เฉพาะ

---

### 2. Application Delivery-only Environments
- มุ่งเน้นการส่งมอบแอปพลิเคชัน
- รองรับ:
  - On-demand Scaling
  - ระบบความปลอดภัยของแอปพลิเคชัน

---

### 3. Open Platform as a Service
- เป็น PaaS แบบ Open Source
- ช่วยให้ผู้ให้บริการสามารถนำไปใช้รันแอปพลิเคชันได้อย่างยืดหยุ่น

---

### 4. Add-on Development Facilities
- เป็น PaaS ที่ใช้เสริมบน SaaS เดิม
- ช่วยให้สามารถปรับแต่งหรือเพิ่มความสามารถให้แพลตฟอร์มที่มีอยู่แล้ว
